# Аудит веб-проекта: техническая оптимизация, SEO и производительность

Дата аудита: **2026-03-25**  
Формат: статический технический аудит репозитория + проверка доступности публичного домена из среды CI.

## 1) Executive summary

Проект уже имеет крепкий базовый SEO-фундамент: корректные `title/description/canonical`, микроразметка `RealEstateAgent`, sitemap/robots, ЧПУ и 301-редиректы.

Главные риски для мобильной скорости и Google-трафика:

1. **Большой критический CSS-пакет** на главной (bootstrap + style ≈ 349 KB raw).
2. **Тяжёлый JS на старте**, включая сторонние виджеты и Chart.js.
3. **Неравномерная работа с изображениями** (часть `<img>` без width/height, отсутствует `srcset/sizes` на ряде страниц).
4. **Проблемные сигналы кэширования third-party и cache busting через `Date.now()`** для формы.
5. **Местами слабая семантика заголовков** (в т.ч. 2×`h1` в `blog-detail.html`, ранние `h6` до `h1` на `index.html`).

Если приоритизировать мобильную производительность под Google, сначала стоит сделать: 
**(P1)** критический CSS + разгрузка JS + lazy/conditional third-party, затем **(P2)** изображение/LCP/CLS и усиление on-page SEO для коммерческих страниц.

## 2) Сильные стороны текущего состояния

### SEO/техническая индексация

- На ключевых страницах есть базовые SEO-теги: `title`, `description`, `canonical`.
- Главная использует корректные OpenGraph/Twitter-теги.
- Есть JSON-LD (`RealEstateAgent`) с контактами, гео и соц-профилями.
- Настроен `robots.txt` + указан `Sitemap`.
- Есть `sitemap.xml` с URL, `lastmod`, `changefreq`, `priority`.
- Настроены 301-редиректы на HTTPS + canonical-домен, удаление `index.html`, legacy URL → новые ЧПУ.

### Производительность/техбаза

- Включён GZIP (`mod_deflate`) для html/css/js/json/svg.
- Есть заголовки кэширования для CSS/JS/изображений/шрифтов.
- На главной применён `defer` для основной части скриптов.
- Есть отложенная загрузка части CSS через `preload + onload`.
- Шрифты подключены с `font-display: swap`.
- Многие изображения уже в WebP.

## 3) Слабые стороны и риски

## 3.1 Производительность (особенно mobile)

### A. Большой объём стартовых CSS/JS

По локальным файлам главной:

- `css/bootstrap.min.css` ≈ **186.5 KB raw**
- `css/style.css` ≈ **162.5 KB raw**
- `js/swiper-bundle.min.js` ≈ **151.6 KB raw**
- `js/optimize.js` ≈ **29.6 KB raw**

Даже с gzip это ощутимо на мобильных сетях с высоким RTT. Для главной, ориентированной на лидогенерацию, это влияет на **LCP/INP**.

### B. Третьи стороны в ранней фазе

- Messenger-виджет грузится `defer` уже на первом экране.
- Форма грузится после `load`, но URL с `?v=Date.now()` фактически отключает кэш на каждый визит.

Это повышает нестабильность TTI/INP и сетевой шум.

### C. Изображения и CLS

На ряде страниц есть `<img>` без пары `width/height`, что может вызывать layout shift.

### D. Тяжёлые библиотеки, не всегда критичные

- `Chart.js` подключается на главной, даже если блок не в первом пользовательском сценарии.
- Нет признаков условной загрузки по `IntersectionObserver`/по факту взаимодействия.

## 3.2 SEO структура и контент

### A. Заголовочная структура

- В `blog-detail.html` найдено **2 h1**.
- На `index.html` встречаются `h6` до основного `h1` (семантически шумно).

Это не «фатал», но ухудшает ясность документа для поисковых алгоритмов и accessibility.

### B. Каноникал и единообразие

- В `services-detail.html` отсутствует canonical.
- Разные страницы частично разнородны по SEO-шаблону head-блока.

### C. Ключевые слова и интент

По титлам/описаниям/`h1` выраженно доминируют запросы уровня:

- «риэлтер в Лиде»
- «покупка/продажа квартир»

Но меньше покрыты коммерческие long-tail намерения (оценка, ипотека, срочный выкуп, проверка юридической чистоты, районные кластеры и т.д.), что ограничивает расширение органики.

## 3.3 UX/конверсия

- Возможная перегрузка первого экрана внешними скриптами может замедлять интерактивность на телефонах.
- На листинговых/детальных страницах полезно усилить «внутреннюю навигацию по намерениям» (FAQ, сравнения районов, перелинковка по сегментам), чтобы повышать глубину просмотра и удержание.

## 4) Приоритетный план улучшений (с внедрением)

## P1 — сделать в первую очередь (влияние на mobile speed + Google)

### 1. Разделить и облегчить критический CSS

**Цель:** снизить render-blocking и улучшить LCP/FCP.

**Что делать:**
1. Выделить критические стили above-the-fold (header + hero + первый CTA) в `critical-home.css` (8–15 KB gzip).
2. Оставить в `<head>` только `critical-home.css` + минимальный reset.
3. Остальные стили грузить как non-critical (`preload`→`stylesheet` или `media="print"` switch).
4. Удалить неиспользуемые Bootstrap-компоненты (перейти на кастомный build).

**Ожидаемый эффект:** заметный прирост mobile LCP и скорости первого рендера.

---

### 2. Условная загрузка JS и third-party

**Цель:** улучшить INP/TBT и устойчивость мобильного UX.

**Что делать:**
1. Перенести загрузку Messenger-виджета на событие взаимодействия (`click`, `scroll`, `pointerdown`) или через idle timeout 4–8 сек.
2. Для формы убрать `Date.now()` в URL; заменить на версионирование при релизе (`forms.js?v=2026-03-25-1`).
3. `Chart.js` грузить только если блок аналитики реально попал в viewport (через `IntersectionObserver`).
4. Проверить `optimize.js`: разделить на критические/некритические куски.

**Ожидаемый эффект:** меньше main-thread блокировок, лучше отзывчивость на мобильных.

---

### 3. LCP/CLS контроль изображений

**Цель:** стабилизировать визуальную загрузку и ускорить hero.

**Что делать:**
1. Для всех `<img>` задать `width` и `height` (или `aspect-ratio` + фиксированные контейнеры).
2. Для hero использовать `fetchpriority="high"` только на 1 реальном LCP-изображении.
3. Добавить `srcset/sizes` для карточек/листингов и адаптивных блоков.
4. Перепроверить вес hero-картинки: целиться в ~60–90 KB для mobile (без видимого деграда).

**Ожидаемый эффект:** ниже CLS, быстрее визуальное появление контента.

---

## P2 — после P1

### 4. Стандартизировать SEO-шаблон head на всех страницах

**Что делать:**
1. Ввести единый шаблон для `title/description/canonical/og/twitter`.
2. Добавить canonical в `services-detail.html`.
3. Проверить уникальность title/description по всем landing/detail страницам.

---

### 5. Улучшить структуру заголовков и on-page семантику

**Что делать:**
1. На странице оставлять **один** `h1`.
2. Убрать «декоративные» h-теги из несемантических мест (заменить на `div/span` + CSS).
3. Собрать кластерную структуру `h2` под коммерческий интент:
   - покупка квартиры в Лиде
   - продажа квартиры в Лиде
   - юридическое сопровождение сделки
   - срочная продажа/подбор

---

### 6. Расширить семантическое ядро под Google long-tail

**Что делать:**
1. Подготовить контент-кластеры под интенты:
   - «купить квартиру в Лиде [район]»
   - «продать квартиру в Лиде быстро»
   - «проверка документов на недвижимость Беларусь»
2. Для каждого кластера — отдельные SEO-страницы/статьи + перелинковка на объекты.
3. Добавить FAQ-блоки с вопросами из реальных интентов и разметкой `FAQPage` там, где это уместно.

---

## P3 — инфраструктура и мониторинг

### 7. Перейти с только GZIP на Brotli (если хостинг поддерживает)

**Что делать:**
1. Включить Brotli для текстовых ресурсов.
2. Проверить `Content-Encoding` для HTML/CSS/JS.

---

### 8. Улучшить cache strategy и валидацию

**Что делать:**
1. Для статики использовать immutable-стратегию (версионированные имена файлов/хэши).
2. Для HTML оставить короткий TTL + revalidate.
3. Для JSON API определить SLA по обновлению и соответствующий `Cache-Control`.

---

### 9. Внедрить регулярный контроль Core Web Vitals

**Что делать:**
1. Запустить еженедельный прогон Lighthouse (mobile).
2. Подключить RUM-метрики (например, через web-vitals + GA4/BigQuery).
3. Установить пороги алертов:
   - LCP < 2.5s (75-й перцентиль)
   - INP < 200ms
   - CLS < 0.1

## 7) Ограничения текущего аудита

Из данной среды публичный URL `https://turko.by` возвращал `403` на HEAD/GET, поэтому полевые метрики PSI/CrUX и live waterfall из внешней проверки получить не удалось.

Поэтому выводы по скорости основаны на:
- локальном составе HTML/CSS/JS и их весе;
- текущих серверных правилах в `.htaccess`;
- структуре SEO-документов и семантики страниц.
