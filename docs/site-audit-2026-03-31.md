# Аудит веб-проекта turko.by (31.03.2026)

## 1) Сильные стороны

- На ключевых страницах корректно заданы `title`, `description`, `canonical`, OpenGraph и Twitter-мета.
- На главной есть структурированные данные `RealEstateAgent` (JSON-LD).
- Для главного изображения используется `preload` + `fetchpriority="high"`.
- Основные скрипты подключены через `defer`, изображения в карточках и слайдерах в основном имеют `loading="lazy"`.
- Есть `robots.txt` и `sitemap.xml`.
- В админке уже реализованы полезные метрики портфеля (ликвидность, экспозиция, стагнация и т.д.).

## 2) Технические и SEO-проблемы

### Критичный приоритет (P1)

1. **Некорректные шаблонные SEO-мета на динамических страницах**
   - `object-detail.html` имеет дефолтный `canonical` на главную страницу и общий `title`.
   - `services-detail.html` использует нецелевой шаблонный `title` (про блог).
   - Риск: до выполнения JS поисковик может зафиксировать нерелевантные мета.

2. **Версионирование ресурсов отключает кэш браузера**
   - `site-version.php` выставляет `window.SITE_VERSION = time()` на каждый запрос.
   - Это приводит к постоянному изменению URL у версионируемых ассетов и фактически ломает long-cache.

3. **`fetch(..., { cache: "no-store" })` на данных объектов**
   - В `js/object-detail.js` для JSON применён `no-store`.
   - Риск: повторные посещения страниц/переходы по объектам каждый раз тянут данные заново.

### Высокий приоритет (P2)

4. **Дубли/неконсистентность заголовков H1 на blog-detail**
   - В `blog-detail.html` присутствует 2 элемента `h1`.

5. **Тяжёлые CSS/шрифты без тонкой оптимизации**
   - `css/style.css` ~162 KB, webfont-иконки (`fa-solid`, `fa-brands`) > 90 KB каждый.

6. **Недостаточно явной серверной политики кэширования и сжатия**
   - В репозитории нет конфигурации заголовков `Cache-Control`, `ETag`, `Brotli/Gzip`.

### Средний приоритет (P3)

7. **Семантика карточек и внутренних ссылок может быть усилена**
   - Потенциал роста по внутренней перелинковке: районы/улицы/типы/ценовые сегменты.

8. **Недостаточная глубина контентных кластеров под локальный спрос**
   - Ядро по частотным запросам «квартира в Лиде», «дом в Лиде», «риэлтер Лида» есть, но мало посадочных страниц под микроинтент.

## 3) Приоритетный roadmap внедрения

## Шаг 1 (P1): стабилизировать SEO-шаблоны и canonical

### Что сделать
- В `object-detail.html` задать безопасный шаблонный canonical (`/object`) и нейтральный title/description для SSR-фазы.
- В `services-detail.html` заменить title/description/canonical на сервисные.
- В `blog-detail.html` оставить один `h1`, второй перевести в `h2`/`div`.

### Как внедрить
1. Подготовить единый helper для мета-обновления (head manager).
2. Прогнать проверку всех `*.html` на наличие: 1x title, 1x description, 1x canonical, 1x h1.
3. Проверить URL в sitemap, чтобы они совпадали с canonical-шаблонами.

### KPI
- 0 страниц с дублирующим `h1`.
- 0 страниц с canonical на нерелевантный URL.

## Шаг 2 (P1): включить эффективное кэширование ассетов

### Что сделать
- Перевести `site-version.php` с `time()` на стабильную версию деплоя (`filemtime(.deploy)` или git hash).
- Для статических файлов (`.css`, `.js`, `.webp`, `.woff2`) включить `Cache-Control: public, max-age=31536000, immutable`.

### Как внедрить
1. Изменить генератор версии на детерминированный.
2. На веб-сервере добавить правила кэша для статических расширений.
3. Для HTML оставить короткий TTL (например 5–15 мин).

### KPI
- Повторные визиты загружают CSS/JS из memory/disk cache.
- Снижение transfer size на повторных просмотрах >70%.

## Шаг 3 (P1): убрать `no-store` для объектных JSON

### Что сделать
- В `js/object-detail.js` заменить `cache: "no-store"` на `default`/`force-cache` (или версионируемый URL JSON).

### Как внедрить
1. Для `objects-list.json` использовать кэширование на 5–30 мин.
2. Для individual `objects/*.json` — 5 мин + revalidate.
3. При обновлении данных инвалидация через версию файла.

### KPI
- Сокращение времени загрузки карточки объекта при повторном открытии.
- Снижение количества сетевых запросов к JSON.

## Шаг 4 (P2): снизить вес CSS/JS

### Что сделать
- Выделить critical CSS по шаблонам (home, list, detail).
- Перейти с full FontAwesome на subset (или SVG sprite).
- Объединить мелкие скрипты, разделить тяжелые модули по route-based lazy load.

### Как внедрить
1. Запустить coverage в DevTools, зафиксировать неиспользуемый CSS.
2. Сформировать отдельный `critical-home.css`.
3. Для неключевых секций применять `content-visibility: auto`.

### KPI
- Уменьшение CSS payload минимум на 30–40%.
- Улучшение LCP и TBT на мобильных устройствах.

## Шаг 5 (P2/P3): развить SEO-кластеры

### Что сделать
- Посадочные страницы: «1-комнатные в Лиде», «квартиры до X USD», «дома в Лидском районе», «новостройки/вторичка». 
- FAQ-кластеры под long-tail: сроки продажи, налоги, документы, ипотека.
- Внутренняя перелинковка между объектами, блогом и FAQ.

### Как внедрить
1. Собрать кластерную матрицу запросов.
2. Связать каждый кластер с отдельной страницей + schema FAQ/Article.
3. Публиковать 2–4 материала в месяц с внутренними ссылками на объекты.

### KPI
- Рост органических входов по НЧ/СЧ.
- Рост глубины просмотра и CTR сниппетов.

## 4) Рекомендации для админки (аналитика объектов)

1. **Воронка объекта**: показы → сохранения → лиды → показы квартиры → сделка.
2. **SLA по лидам**: среднее время ответа, доля ответов <15 мин.
3. **Heatmap спроса**: районы/улицы Лиды, где выше CTR/конверсия.
4. **Price Intelligence**: сигнал «цена выше рынка на X%», автоподсказка диапазона торга.
5. **Рекомендатор действий**: «снизить цену на 3–5%», «заменить первые 3 фото», «усилить описание».
6. **A/B блоков карточки**: разные первые фото/CTA/порядок параметров.
7. **Прогноз срока продажи**: модель на основе цены, ликвидности, экспозиции, локации.
8. **Контроль качества данных**: детект пропусков (площадь, этаж, год, фото < N).
9. **Календарь активности**: напоминания обновить объект, переснять фото, пересмотреть цену.
10. **Сводка по источникам лидов**: Telegram/Instagram/Direct/органика с CAC и CR.

## 5) Современные UI/UX решения для сайта риэлтера (Лида)

1. **Карточки объектов с sticky CTA** (Позвонить / Telegram / Записаться на просмотр).
2. **Smart filters 2.0**: бюджет-ползунок, время до центра, этажность, «только с балконом/ремонтом».
3. **Сравнение объектов side-by-side** (до 3 объектов).
4. **Микроанимации статусов** (новый/снижение цены/срочно), но без перегруза.
5. **«Похожие объекты» по ML-правилам** (цена, район, метраж, ликвидность).
6. **Локальный контент-блок «Районы Лиды»** с картой, инфраструктурой и динамикой цен.
7. **Калькулятор сценариев покупки**: первый взнос, платеж, прогноз перепродажи.
8. **Прозрачный trust-блок**: реальные кейсы, таймлайн сделки, документы и лицензия.
9. **One-click лид-формы** с автоподстановкой объекта и удобным мессенджером.
10. **UX для мобильных**: fixed bottom bar, крупные цели касания, ускоренный flow заявки (до 3 полей).

## 6) Базовые метрики контроля после внедрения

- Core Web Vitals: LCP, INP, CLS (mobile first).
- SEO: индексация, coverage, CTR, позиции по кластерам «Лида + тип недвижимости».
- Business KPI: CR лидов, скорость ответа, срок экспозиции, доля проданных без снижения цены.
