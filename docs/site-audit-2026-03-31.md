# Аудит веб-проекта turko.by (31.03.2026)

## 1) Сильные стороны

- На ключевых страницах корректно заданы `title`, `description`, `canonical`, OpenGraph и Twitter-мета.
- На главной есть структурированные данные `RealEstateAgent` (JSON-LD).
- Для главного изображения используется `preload` + `fetchpriority="high"`.
- Основные скрипты подключены через `defer`, изображения в карточках и слайдерах в основном имеют `loading="lazy"`.
- Есть `robots.txt` и `sitemap.xml`.
- В админке уже реализованы полезные метрики портфеля (ликвидность, экспозиция, стагнация и т.д.).

## 2) Технические и SEO-проблемы

### Критичный приоритет (P1)

1. **Некорректные шаблонные SEO-мета на динамических страницах**
   - `object-detail.html` имеет дефолтный `canonical` на главную страницу и общий `title`.
   - `services-detail.html` использует нецелевой шаблонный `title` (про блог).
   - Риск: до выполнения JS поисковик может зафиксировать нерелевантные мета.

2. **Версионирование ресурсов отключает кэш браузера**
   - `site-version.php` выставляет `window.SITE_VERSION = time()` на каждый запрос.
   - Это приводит к постоянному изменению URL у версионируемых ассетов и фактически ломает long-cache.

3. **`fetch(..., { cache: "no-store" })` на данных объектов**
   - В `js/object-detail.js` для JSON применён `no-store`.
   - Риск: повторные посещения страниц/переходы по объектам каждый раз тянут данные заново.

### Высокий приоритет (P2)

4. **Дубли/неконсистентность заголовков H1 на blog-detail**
   - В `blog-detail.html` присутствует 2 элемента `h1`.

5. **Тяжёлые CSS/шрифты без тонкой оптимизации**
   - `css/style.css` ~162 KB, webfont-иконки (`fa-solid`, `fa-brands`) > 90 KB каждый.

6. **Недостаточно явной серверной политики кэширования и сжатия**
   - В репозитории нет конфигурации заголовков `Cache-Control`, `ETag`, `Brotli/Gzip`.

### Средний приоритет (P3)

7. **Семантика карточек и внутренних ссылок может быть усилена**
   - Потенциал роста по внутренней перелинковке: районы/улицы/типы/ценовые сегменты.

8. **Недостаточная глубина контентных кластеров под локальный спрос**
   - Ядро по частотным запросам «квартира в Лиде», «дом в Лиде», «риэлтер Лида» есть, но мало посадочных страниц под микроинтент.

## 3) Приоритетный roadmap внедрения

## Шаг 1 (P1): стабилизировать SEO-шаблоны и canonical

### Что сделать
- В `object-detail.html` задать безопасный шаблонный canonical (`/object`) и нейтральный title/description для SSR-фазы.
- В `services-detail.html` заменить title/description/canonical на сервисные.
- В `blog-detail.html` оставить один `h1`, второй перевести в `h2`/`div`.

### Как внедрить
1. Подготовить единый helper для мета-обновления (head manager).
2. Прогнать проверку всех `*.html` на наличие: 1x title, 1x description, 1x canonical, 1x h1.
3. Проверить URL в sitemap, чтобы они совпадали с canonical-шаблонами.

### KPI
- 0 страниц с дублирующим `h1`.
- 0 страниц с canonical на нерелевантный URL.

## Шаг 2 (P1): включить эффективное кэширование ассетов

### Что сделать
- Перевести `site-version.php` с `time()` на стабильную версию деплоя (`filemtime(.deploy)` или git hash).
- Для статических файлов (`.css`, `.js`, `.webp`, `.woff2`) включить `Cache-Control: public, max-age=31536000, immutable`.

### Как внедрить
1. Изменить генератор версии на детерминированный.
2. На веб-сервере добавить правила кэша для статических расширений.
3. Для HTML оставить короткий TTL (например 5–15 мин).

### KPI
- Повторные визиты загружают CSS/JS из memory/disk cache.
- Снижение transfer size на повторных просмотрах >70%.

## Шаг 3 (P1): убрать `no-store` для объектных JSON

### Что сделать
- В `js/object-detail.js` заменить `cache: "no-store"` на `default`/`force-cache` (или версионируемый URL JSON).

### Как внедрить
1. Для `objects-list.json` использовать кэширование на 5–30 мин.
2. Для individual `objects/*.json` — 5 мин + revalidate.
3. При обновлении данных инвалидация через версию файла.

### KPI
- Сокращение времени загрузки карточки объекта при повторном открытии.
- Снижение количества сетевых запросов к JSON.

## Шаг 4 (P2): снизить вес CSS/JS

### Что сделать
- Выделить critical CSS по шаблонам (home, list, detail).
- Перейти с full FontAwesome на subset (или SVG sprite).
- Объединить мелкие скрипты, разделить тяжелые модули по route-based lazy load.

### Как внедрить
1. Запустить coverage в DevTools, зафиксировать неиспользуемый CSS.
2. Сформировать отдельный `critical-home.css`.
3. Для неключевых секций применять `content-visibility: auto`.

### KPI
- Уменьшение CSS payload минимум на 30–40%.
- Улучшение LCP и TBT на мобильных устройствах.

## Шаг 5 (P2/P3): развить SEO-кластеры

### Что сделать
- Посадочные страницы: «1-комнатные в Лиде», «квартиры до X USD», «дома в Лидском районе», «новостройки/вторичка». 
- FAQ-кластеры под long-tail: сроки продажи, налоги, документы, ипотека.
- Внутренняя перелинковка между объектами, блогом и FAQ.

### Как внедрить
1. Собрать кластерную матрицу запросов.
2. Связать каждый кластер с отдельной страницей + schema FAQ/Article.
3. Публиковать 2–4 материала в месяц с внутренними ссылками на объекты.

### KPI
- Рост органических входов по НЧ/СЧ.
- Рост глубины просмотра и CTR сниппетов.

## 4) Рекомендации для админки (аналитика объектов)

1. **Воронка объекта**: показы → сохранения → лиды → показы квартиры → сделка.
2. **SLA по лидам**: среднее время ответа, доля ответов <15 мин.
3. **Heatmap спроса**: районы/улицы Лиды, где выше CTR/конверсия.
4. **Price Intelligence**: сигнал «цена выше рынка на X%», автоподсказка диапазона торга.
5. **Рекомендатор действий**: «снизить цену на 3–5%», «заменить первые 3 фото», «усилить описание».
6. **A/B блоков карточки**: разные первые фото/CTA/порядок параметров.
7. **Прогноз срока продажи**: модель на основе цены, ликвидности, экспозиции, локации.
8. **Контроль качества данных**: детект пропусков (площадь, этаж, год, фото < N).
9. **Календарь активности**: напоминания обновить объект, переснять фото, пересмотреть цену.
10. **Сводка по источникам лидов**: Telegram/Instagram/Direct/органика с CAC и CR.

## 5) Современные UI/UX решения для сайта риэлтера (Лида)

1. **Карточки объектов с sticky CTA** (Позвонить / Telegram / Записаться на просмотр).
2. **Smart filters 2.0**: бюджет-ползунок, время до центра, этажность, «только с балконом/ремонтом».
3. **Сравнение объектов side-by-side** (до 3 объектов).
4. **Микроанимации статусов** (новый/снижение цены/срочно), но без перегруза.
5. **«Похожие объекты» по ML-правилам** (цена, район, метраж, ликвидность).
6. **Локальный контент-блок «Районы Лиды»** с картой, инфраструктурой и динамикой цен.
7. **Калькулятор сценариев покупки**: первый взнос, платеж, прогноз перепродажи.
8. **Прозрачный trust-блок**: реальные кейсы, таймлайн сделки, документы и лицензия.
9. **One-click лид-формы** с автоподстановкой объекта и удобным мессенджером.
10. **UX для мобильных**: fixed bottom bar, крупные цели касания, ускоренный flow заявки (до 3 полей).

## 6) Базовые метрики контроля после внедрения

- Core Web Vitals: LCP, INP, CLS (mobile first).
- SEO: индексация, coverage, CTR, позиции по кластерам «Лида + тип недвижимости».
- Business KPI: CR лидов, скорость ответа, срок экспозиции, доля проданных без снижения цены.

## 7) Подробный план внедрения SEO-кластеров (пошагово)

Ниже — практический сценарий, как развернуть кластерную SEO-структуру без перегруза команды.

### Этап 0. Подготовка (1–2 дня)

1. Зафиксировать целевые KPI на 3 месяца:
   - +30–50% органики по НЧ/СЧ.
   - +20% заявок из органики.
   - не менее 30% материалов с CTR > 4%.
2. Назначить ответственных:
   - SEO/контент;
   - редактор/копирайтер;
   - разработчик (шаблоны + schema + перелинковка).
3. Подготовить таблицу-контур (Google Sheets/Notion):
   - `кластер`, `страница`, `интент`, `ключи`, `URL`, `статус`, `дата публикации`, `внутренние ссылки`.

### Этап 1. Кластерная матрица запросов (3–5 дней)

Соберите запросы в 5 групп:

1. **Комнатность**:  
   «1-комнатные квартиры в Лиде», «2-комнатные в Лиде», «3-комнатные в Лиде».
2. **Бюджет**:  
   «квартиры в Лиде до 30000 usd», «до 40000 usd».
3. **Тип недвижимости**:  
   «дома в Лидском районе», «новостройки Лида», «вторичка Лида».
4. **Сервисные/транзакционные**:  
   «риэлтер Лида продажа квартиры», «оценка квартиры Лида».
5. **Информационные (FAQ/блог)**:  
   «какие документы нужны для продажи квартиры в Беларуси», «налог при продаже квартиры», «ипотека в Беларуси условия».

Для каждого запроса отметьте:
- частотность (условно: H/M/L),
- конкуренцию,
- тип интента: купить / продать / узнать / сравнить,
- приоритет (P1/P2/P3).

### Этап 2. Карта посадочных страниц (2–3 дня)

Создайте целевые URL-шаблоны:

- `/kvartiry-lida/1-komnatnye`
- `/kvartiry-lida/do-30000-usd`
- `/doma/lidskiy-rayon`
- `/novostroyki-lida`
- `/vtorichka-lida`

Рекомендации по структуре страницы кластера:
1. `H1` с главным ключом.
2. SEO-текст 1200–2500 знаков (не «простыня», а блоками).
3. Блок «Актуальные объекты» (карточки, подгружаемые из `objects-list.json`).
4. Блок «Частые вопросы» (3–6 вопросов).
5. CTA: консультация, звонок, Telegram/Viber.
6. Ссылки на смежные кластеры (например, из «1-комнатные» в «до 30000 USD»).

### Этап 3. FAQ-кластеры и long-tail (еженедельно)

Соберите 20–40 вопросов и распределите:

- 8–12 вопросов на отдельную `FAQ`-страницу;
- 10–20 вопросов по тематическим статьям в блоге;
- 3–5 FAQ внутри каждой коммерческой страницы кластера.

Правило контента FAQ:
1. вопрос в форме реального поискового запроса;
2. короткий ответ (400–800 символов);
3. конкретика (сроки, документы, налоги);
4. в конце — ссылка на связанный объект или услугу.

### Этап 4. Внутренняя перелинковка (обязательно на каждой публикации)

Для каждой новой страницы/статьи:
1. Добавить 2–3 ссылки на релевантные кластеры.
2. Добавить 2 ссылки на конкретные объекты.
3. Добавить 1 ссылку на профильную услугу.
4. Добавить 1 ссылку на FAQ.

Минимальный шаблон перелинковки:
- **Из блога** → в кластеры + объекты.
- **Из кластеров** → в объекты + FAQ + услуги.
- **Из карточки объекта** → в похожие объекты + тематический блог + FAQ по сделке.

### Этап 5. Schema-разметка (техническая реализация)

1. Для посадочных и FAQ-блоков:
   - `FAQPage` (если есть список вопросов/ответов);
   - `BreadcrumbList` на всех кластерных страницах.
2. Для статей:
   - `Article` или `BlogPosting`.
3. Для карточек объектов:
   - `Offer` + `BreadcrumbList` (у вас уже частично реализовано).

Проверка:
- валидировать в Rich Results Test;
- исправлять предупреждения в первую очередь на P1-страницах.

### Этап 6. Контент-план 2–4 публикации/месяц (на 12 недель)

Рекомендуемый ритм:
- Неделя 1: коммерческая кластерная страница.
- Неделя 2: FAQ-материал.
- Неделя 3: обзор рынка/района.
- Неделя 4: кейс сделки (до/после, срок экспозиции, стратегия цены).

Пример 1 месяца:
1. «1-комнатные квартиры в Лиде: что реально купить в 2026».
2. «Какие документы нужны для продажи квартиры в Беларуси в 2026».
3. «Районы Лиды: где быстрее продаются квартиры».
4. «Кейс: продали квартиру за 21 день без дисконта».

### Этап 7. Контроль качества перед публикацией (чек-лист)

Перед релизом каждой страницы проверять:
1. уникальные `title`, `description`, `canonical`;
2. ровно один `h1`;
3. заполнены alt у ключевых изображений;
4. есть 3–6 внутренних ссылок;
5. есть schema;
6. страница добавлена в sitemap;
7. URL отправлен в индексирование (Search Console / Яндекс Вебмастер).

### Этап 8. Мониторинг и итерации (каждые 2 недели)

Смотрите по каждой странице:
- показы, клики, CTR, средняя позиция;
- глубина просмотра;
- конверсия в заявку;
- доля отказов.

Правило оптимизации:
- если позиция 8–20 и CTR низкий — доработать `title/description`;
- если трафик есть, а лидов мало — усилить CTA и блоки доверия;
- если нет роста 6–8 недель — расширить семантику и добавить ссылки из сильных страниц.

### Этап 9. KPI для SEO-кластеров (что и как измерять)

Ниже — рабочий набор KPI по воронке: индексация → трафик → поведение → лиды.

#### 9.1 Индексация и техническое здоровье

1. **Index Coverage (кластерные URL)**
   - Формула: `индексируемые кластерные страницы / опубликованные кластерные страницы × 100%`.
   - Цель: **не ниже 95%** через 30 дней после публикации.
2. **Доля страниц без ошибок schema**
   - Формула: `страницы без ошибок Rich Results / страницы со schema × 100%`.
   - Цель: **100% без ошибок**, предупреждения ≤ 20%.
3. **Скорость индексации**
   - Формула: среднее число дней от публикации до появления в индексе.
   - Цель: **до 7 дней** для новых страниц.

#### 9.2 Видимость и кликабельность в поиске

4. **Рост показов по кластерам**
   - Формула: `(показы текущего периода - показы базового периода) / показы базового периода × 100%`.
   - Цель: **+30–50% за 3 месяца**.
5. **Средняя позиция P1-кластеров**
   - Цель: вывести приоритетные URL в **ТОП-10** за 2–4 месяца.
6. **CTR кластерных страниц**
   - Цель: **4–8%** (по long-tail обычно выше; по широким ключам может быть ниже).

#### 9.3 Поведенческие KPI

7. **Глубина просмотра из SEO-кластеров**
   - Формула: среднее число просмотренных страниц за сессию, начатую с кластерного URL.
   - Цель: **≥ 2.2** страницы/сессию.
8. **Доля переходов из кластера в карточки объектов**
   - Формула: `сессии с переходом в объект / все сессии на кластере × 100%`.
   - Цель: **20–35%**.
9. **Bounce Rate кластерных страниц**
   - Цель: держать **< 55%** (для информационных FAQ допустимо выше).

#### 9.4 Бизнес-KPI (лиды и сделки)

10. **CR в лид с органики (кластерные URL)**
    - Формула: `лиды из SEO-кластеров / органические сессии на кластерах × 100%`.
    - Цель: **2–5%** (зависит от качества трафика и CTA).
11. **Скорость реакции на лид**
    - Цель: медиана первого ответа **≤ 15 минут**.
12. **Доля лидов, дошедших до показа**
    - Цель: **≥ 25%** от лидов из SEO-канала.

### Этап 10. План внедрения KPI по неделям (12 недель)

#### Недели 1–2
- Настроить события аналитики:
  - `cluster_page_view`
  - `cluster_to_object_click`
  - `lead_submit`
  - `messenger_click`
- Разметить UTM для внутренних CTA-переходов (или отдельные event params).
- Собрать baseline по показам/CTR/позициям.

#### Недели 3–4
- Запустить 2 первые кластера + 2 FAQ материала.
- Проверить индексацию, schema, внутренние ссылки.
- Снять первый срез: показы, CTR, переходы в объекты.

#### Недели 5–8
- Дублировать успешный шаблон на 3–5 новых кластеров.
- Для страниц с позицией 8–20: переписать сниппеты (`title`/`description`) и добавить FAQ-блок.
- Усилить перелинковку со «сильных» страниц (где есть трафик).

#### Недели 9–12
- Оценить CR в лид по каждому кластеру.
- Отключить/объединить слабые кластеры без роста.
- Перераспределить контент-план в пользу тем с лучшим `cluster_to_object_click` и `lead_submit`.

### Этап 11. KPI-дашборд (минимальный состав)

Соберите единый дашборд (Looker Studio / Power BI / Metabase) с 4 вкладками:

1. **Index & Tech**: индексация, schema, скорость индексации.
2. **Search Performance**: показы, клики, CTR, позиции (по кластерам).
3. **Behavior**: глубина, bounce, переходы в объекты.
4. **Leads**: заявки, CR, SLA ответа, конверсия в показы.

Минимальная частота ревью:
- еженедельно — оперативные метрики (трафик/CTR/лиды),
- раз в 2 недели — решения по контенту и SEO-итерациям,
- ежемесячно — ревизия кластерной матрицы и KPI-целей.
