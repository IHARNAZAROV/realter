# UI/UX блоки для роста лидов в недвижимости (аудит + новые идеи)

Дата: 2026-04-06

## Что уже реализовано (по текущему сайту)

1. Каталог с фильтрами и сортировкой (тип, комнаты, цена, локация), счётчик объектов, избранное, lazy "Показать ещё".
2. Lead Capture для пустой выдачи: форма контакта (Telegram/телефон) + передача фильтров на backend.
3. Запись на просмотр объекта: модальное окно, календарь, выбор времени, валидация телефона, отправка заявки.
4. Ипотечный калькулятор и выбор программ нескольких банков.
5. Блок рыночной аналитики (средние цены, средняя цена за м², тренд, график).
6. Модальный чеклист документов по типу сделки (покупка/продажа/обмен) + прогресс + отправка в Telegram.

## Какие блоки можно добавить (без дублирования текущих)

### Приоритет A (самый быстрый рост конверсии)

1. **Квиз подбора “за 60 секунд” (intent capture до фильтров)**
   - 5–7 шагов: бюджет, район, тип, срок переезда, ипотека/наличные.
   - Финал: “получить 3 подходящих варианта в Telegram/WhatsApp”.
   - Почему: сейчас лид появляется в основном при *пустой выдаче*; квиз даст лиды ещё до этого этапа.

2. **Sticky multi-CTA панель на мобильном**
   - Кнопки: Позвонить / Написать в Telegram / Оставить заявку.
   - Плавающая панель на карточках объектов и в каталоге.
   - Почему: сокращает путь к контакту с 2–3 кликов до 1.

3. **Exit-intent / scroll-intent оффер**
   - На десктопе: при попытке ухода.
   - На мобиле: после 70–80% скролла без действия.
   - Оффер: “Отправлю свежие объекты, которых ещё нет в ленте”.

4. **Форма “Подберу объект под ваш бюджет” прямо под фильтрами**
   - Имя + контакт + бюджет/район (минимум полей).
   - Использовать как fallback не только при нулевом результате.

### Приоритет B (усиление доверия и качества лида)

5. **Блок “Разбор сделки на примере” (кейсы до/после)**
   - Проблема клиента → решение → сроки → финансовый результат.
   - CTA: “Хочу такой же разбор по моей ситуации”.

6. **Калькулятор полной стоимости покупки (TCO), не только ипотека**
   - Добавить налоги/пошлины/страховки/оценку/сделку/ремонт.
   - Результат: “реальный бюджет входа” + кнопка консультации.

#### Как я вижу TCO-калькулятор (детальный формат)

**Цель блока:** показать клиенту не только ежемесячный платёж, а **полный бюджет входа в сделку**:
1) сколько нужно денег "сейчас",
2) сколько будет ежемесячная нагрузка,
3) какой нужен финансовый буфер.

##### 1) Структура расчёта (3 экрана/шага)

1. **Данные объекта и сценарий покупки**
   - Цена объекта.
   - Тип: вторичка / новостройка.
   - Способ оплаты: наличные / ипотека.
   - Для ипотеки: первоначальный взнос, ставка, срок.

2. **Разовые расходы на входе**
   - Нотариальные/регистрационные расходы.
   - Госпошлины/обязательные сборы.
   - Оценка объекта.
   - Услуги банка (если применимо).
   - Страхование (жизнь/имущество, если требуется банком).
   - Комиссии/сопровождение сделки.
   - Переезд + базовый ремонт/подготовка.

3. **После покупки (первый год)**
   - Ежемесячный платёж по ипотеке (если есть).
   - Коммунальные платежи (ориентир).
   - Плановые расходы на содержание/мелкий ремонт.
   - Резервный фонд (например, 1–3 ежемесячных платежа).

##### 2) Что показывать в результате

- **Итог 1: Нужно на сделку сейчас** (down payment + разовые расходы).
- **Итог 2: Ежемесячная нагрузка** (ипотека + коммунальные + содержание).
- **Итог 3: Рекомендуемый финансовый буфер**.
- **Итог 4: Полная стоимость владения за 12 месяцев**.
- Кнопка CTA: **"Получить персональный разбор с риэлтером"**.

##### 3) UX-подача (чтобы не пугать пользователя)

- Режим **"Быстрый"** (4–5 полей, всё остальное по умолчанию).
- Режим **"Детальный"** (все статьи расходов с ручной настройкой).
- Подсказки "что это?" рядом с каждым пунктом расходов.
- Слайдеры и пресеты:
  - "Эконом-ремонт / Стандарт / Комфорт",
  - "Минимальный / Рекомендуемый буфер".

##### 4) Формулы и логика MVP

- `Вход в сделку = первоначальный взнос + сумма разовых расходов`.
- `Ежемесячная нагрузка = ипотечный платёж + коммунальные + содержание`.
- `Буфер = ежемесячная нагрузка × коэффициент буфера (1..3)`.
- `TCO 12 мес = вход в сделку + (ежемесячная нагрузка × 12) + буфер`.

Для MVP достаточно фиксировать часть ставок как настраиваемые коэффициенты в админке (или в конфиге), без перегруженной юридической детализации на первом этапе.

##### 5) Интеграция в существующий сайт

- На карточке объекта рядом с ипотечным калькулятором добавить кнопку:
  **"Посчитать полный бюджет покупки"**.
- Предзаполнять цену объекта автоматически из карточки.
- Сохранять результаты как лид-событие (`lead_requests`, type=`tco_calc`) при отправке контакта.
- После расчёта предлагать:
  - "Отправить расчёт в Telegram",
  - "Записаться на консультацию",
  - "Подобрать объекты под этот бюджет".

##### 6) Какие поля собирать в лид-форме после расчёта

- Контакт (телефон/Telegram).
- Целевой бюджет "на входе" (сколько готов внести сейчас).
- Комфортный ежемесячный платёж.
- Горизонт сделки (1–3 месяца / 3–6 / позже).

Это сразу повышает качество лида: вы получаете не просто "интерес", а финансово-квалифицированный запрос.

##### 7) Риски и ограничения

- Расчёт должен быть помечен как **ориентировочный**, не оферта.
- Нужен дисклеймер по изменению ставок/тарифов.
- Если пользователь не заполняет детальные поля — использовать рыночные дефолты с прозрачной пометкой.

##### 8) Метрики эффективности именно TCO-блока

- CR: просмотр калькулятора → завершённый расчёт.
- CR: завершённый расчёт → отправка контакта.
- CR: TCO-лид → консультация → показ.
- Доля TCO-лидов в общем пуле лидов.
- Медиана "входного бюджета" и "комфортного платежа" (для сегментации рекламы и контента).

7. **Сравнение объектов side-by-side**
   - До 3 карточек: цена, м², район, этаж, состояние, коммунальные.
   - CTA: “Помочь выбрать лучший вариант”.

8. **Блок “Юридические риски и как я их снимаю”**
   - Частые риски + чек “как проверяю” + гарантия процесса.
   - Хорошо работает для “тёплых, но тревожных” лидов.

### Приоритет C (масштаб и retention)

9. **Подписка на новые объекты по сохранённым фильтрам**
   - Триггер: пользователь применил фильтры.
   - Каналы: Telegram/e-mail.
   - Логика: “сообщим первыми, как только появится”.

10. **Личный мини-кабинет “Мои подборки/избранное/заявки”**
    - Без тяжёлого аккаунта: magic-link по телефону/почте.
    - Повышает возвратные визиты и повторные касания.
    - Что это: персональная страница клиента, где он видит:
      - сохранённые объекты (избранное),
      - историю своих заявок (подбор, просмотр),
      - статусы по заявкам (например: "Принято", "В работе", "Назначен показ"),
      - персональные подборки от риэлтера.

#### Как реализовать мини-кабинет (практичный MVP за 1–2 спринта)

1. **Авторизация без пароля (magic-link / одноразовый код)**
   - Шаг 1: клиент вводит телефон или email.
   - Шаг 2: сервер отправляет одноразовую ссылку (или OTP-код).
   - Шаг 3: по ссылке клиент входит автоматически, создаётся короткая сессия.
   - Почему так: минимальный порог входа и меньше отказов, чем у "логин + пароль".

2. **Структура MVP-данных**
   - `users`: id, phone/email, created_at, last_seen_at
   - `favorites`: user_id, object_id, created_at
   - `lead_requests`: user_id, type (`no_result`, `book_viewing`, `quiz`), payload, status, created_at
   - `agent_collections`: user_id, title, comment, object_ids[], updated_at

3. **Минимальный UI кабинета**
   - Вкладка **Избранное**: карточки + CTA "Назначить просмотр".
   - Вкладка **Мои заявки**: таймлайн действий и текущий статус.
   - Вкладка **Подборки от риэлтера**: наборы объектов с комментариями.
   - Вкладка **Профиль**: контакты и удобный канал связи.

4. **Интеграция с тем, что уже есть на сайте**
   - Кнопка "Добавить в избранное" уже логически присутствует в каталоге — её нужно связать с `user_id`.
   - Формы `lead-no-result` и `book-viewing` уже существуют — нужно сохранять их ещё и в `lead_requests` для отображения статусов в кабинете.
   - События обновления статусов может менять админка (ручной workflow на старте).

5. **Безопасность и UX-ограничения**
   - TTL magic-link: 10–15 минут, одноразовое использование.
   - Rate limit на отправку ссылок/кодов (например, 5 попыток/час на контакт).
   - Маскирование персональных данных в UI (частично скрытый телефон/email).
   - Автовыход при долгой неактивности.

6. **Метрики успеха конкретно для кабинета**
   - Доля пользователей, завершивших вход по magic-link.
   - Retention D7 / D30 среди вошедших в кабинет.
   - Конверсия "избранное → заявка на просмотр".
   - Конверсия "подборка от риэлтера → диалог/звонок".

#### Насколько это сложно реализовать?

- **Короткий ответ:** не "сверхсложно", если делать поэтапно.
- **Оценка сложности MVP:** средняя (≈ 6/10).
- **Оценка сложности full-версии:** выше средней (≈ 8/10), если добавлять глубокую автоматизацию CRM, push-уведомления и расширенную аналитику поведения.

**Почему MVP реалистичен:**
- Базовые лид-формы уже есть (no-results и запись на просмотр), значит часть данных и событий уже собирается.
- Основной объём работы — не "новая математика", а связка существующих сценариев в единый пользовательский контур.
- Passwordless-вход (magic-link/OTP) уменьшает объём UI/логики по сравнению с классической регистрацией.

**Где основная сложность:**
- Безопасность авторизации (одноразовые токены, TTL, rate limit, защита от перебора).
- Согласованность статусов заявок между сайтом и внутренней операционной частью.
- Качественный UX для мобильного потока (быстрый вход, минимум шагов, понятные статусы).

**Реалистичный план по этапам:**
1. **Этап 1 (1 спринт):** вход по magic-link + вкладка "Избранное".
2. **Этап 2 (1 спринт):** вкладка "Мои заявки" + статусы из lead_requests.
3. **Этап 3 (1 спринт):** "Подборки от риэлтера" + базовые метрики retention/conversion.

Итого: рабочий MVP обычно можно собрать за **2–3 спринта**, если не перегружать первую версию интеграциями.

11. **Блок “План сделки по срокам” (timeline)**
    - День 1 → Проверка → Показ → Аванс → Сделка → Передача ключей.
    - Снимает неопределённость и повышает готовность оставить контакт.

12. **Микро-сегментация оффера по типу пользователя**
    - Переключатель: “Покупаю / Продаю / Инвестирую / Переезжаю”.
    - Для каждого — свой оффер и lead form copy.

## Рекомендуемый порядок внедрения

Спринт 1:
- Квиз 60 секунд
- Sticky mobile CTA
- Доп. форма подбора под бюджет

Спринт 2:
- Exit-intent оффер
- TCO-калькулятор
- Кейсы сделок

Спринт 3:
- Подписка на новые объекты
- Сравнение объектов
- Timeline сделки

## Метрики, которые стоит отслеживать

- CR в лид (каталог, карточка объекта, no-results, квиз).
- Доля лидов по источнику блока (%).
- CPL по каждому блоку.
- Скорость первой реакции (SLA) по лидам.
- Конверсия лид → показ → сделка.
