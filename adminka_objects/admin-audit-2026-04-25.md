## Ключевые выводы

### 1) Риск утечки памяти / деградации: `enableDragAndDrop` навешивает обработчики при каждом `render()`
В `render()` каждый раз вызывается `enableDragAndDrop(container, objects)`, а внутри функции заново добавляются `dragstart`, `dragend`, `dragover`, `drop` на тот же контейнер.

**Почему это важно:** после множества рендеров одно действие DnD начнёт триггерить несколько обработчиков одновременно (рост нагрузки + потенциально «дёрганое» поведение).

**Рекомендация:**
- либо ставить флаг `container.__dndBound = true`,
- либо перейти на `AbortController` и пересоздавать подписки контролируемо,
- либо вызывать bind DnD один раз на старте (как уже сделано для других делегированных событий).

### 2) Риск накопления запросов в `direct.js` и `blog-stats.js`
`loadData()`/`loadAll()` вызываются по фильтрам/refresh без отмены предыдущего `fetch`.

**Почему это важно:** при частых кликах (период, кампания, refresh) старые ответы могут приходить позже новых (race condition) и перезаписывать UI устаревшими данными; лишняя нагрузка на сеть/CPU.

**Рекомендация:**
- использовать `AbortController` (хранить `currentRequestController` в state),
- при новом запросе делать `controller.abort()` для предыдущего,
- при рендере проверять `requestId` (token) и игнорировать «просроченные» ответы.

### 3) Частые полные перерисовки через `innerHTML`
В `direct.js` и `blog-stats.js` большие блоки таблиц перерисовываются строками HTML.

**Почему это важно:** на больших наборах данных это заметно грузит GC и layout.

**Рекомендация:**
- сделать «легкую виртуализацию» (постранично/по viewport),
- или рендерить `DocumentFragment` и обновлять только изменившиеся строки,
- для поиска добавить debounce 150–250ms.

### 4) Позитивно (уже хорошо)
- В `admin.js` для основной логики списка уже применено event delegation и есть защита `isInitialized` для одноразового bind — это хороший базис.
- В `json-generator.js` не найдено долгоживущих интервалов/таймеров, только короткие UI-анимации.

## Что добавить в админку: 2 варианта

### Вариант A — «Операционный контроль»
Фокус: скорость ежедневной работы.

- Дашборд ошибок данных (валидация schema + обязательные поля + дубликаты slug/id).
- Очередь изменений (change queue): кто/что изменил, diff до/после, возможность отката 1 клик.
- Массовые операции: bulk-изменение статуса, города, тегов, рекомендованности.
- Профили фильтров (сохранённые пресеты для разных менеджеров).
- SLA-индикаторы: объекты без фото, без цены, «застрявшие» N дней.

### Вариант B — «Аналитика и рост»
Фокус: доход/конверсия и принятие решений.

- Воронка по объектам: публикация → просмотр карточки → заявка → сделка.
- Рекомендатор цены: подсказка диапазона (рынок/ликвидность/экспозиция).
- ABC-классификация объектов по эффективности (трафик/лиды/скорость продажи).
- Автоалерты: «CTR Директа упал >X%», «объект дороже рынка >Y%», «нет действий >Z дней».
- Эксперименты карточек (A/B): заголовок, первое фото, short description.

## Быстрый чек-лист утечек (для браузер-профилирования)

1. Открыть админку, 30–50 раз вызвать `render()` сценариями фильтров/сортировки.
2. В DevTools -> Performance/Memory:
   - следить за ростом `Event Listeners` на `#objectsList`;
   - проверить detached DOM nodes после открытия/закрытия модалок;
   - сравнить heap snapshots «до/после» серии действий.
3. В Network проверить, отменяются ли предыдущие запросы при частом refresh.
