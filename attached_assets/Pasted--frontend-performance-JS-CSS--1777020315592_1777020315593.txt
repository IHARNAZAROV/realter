Ты — опытный frontend performance-инженер.

Задача: провести полный аудит подключения JS и CSS файлов на сайте и определить, какие из них реально используются на каждой странице.

Контекст:
- Сайт написан на vanilla JavaScript (без фреймворков)
- Важно: НЕ менять функционал, только анализ
- Нужно выявить лишние и неиспользуемые ресурсы

Что нужно сделать:

1. Найди ВСЕ HTML-файлы проекта (например: index.html, services-detail.html и т.д.)

2. Для КАЖДОЙ страницы:
   - Определи список подключенных:
     - JS файлов (через <script>)
     - CSS файлов (через <link>)
   
3. Для каждого подключенного файла:
   - Проверить, используется ли он на странице:
     
     ДЛЯ JS:
     - Есть ли вызовы функций из файла
     - Используются ли обработчики событий
     - Есть ли обращения к DOM, которые реально существуют на странице
     - Проверить, не подключается ли файл “на всякий случай”
     
     ДЛЯ CSS:
     - Используются ли классы/селекторы на странице
     - Есть ли стили, которые вообще не применяются
     - Проверить глобальные и page-specific стили

4. Сформировать отчет в следующем формате:

=== PAGE: index.html ===

JS:
- js/main.js → используется (инициализация сайта)
- js/swiper.js → используется (слайдер на главной)
- js/service-detail.js → НЕ используется ❌

CSS:
- css/style.css → используется
- css/swiper.css → используется
- css/admin.css → НЕ используется ❌

5. В конце отчета добавить:

=== ВЫВОД ===

- Список файлов, которые можно удалить с конкретных страниц
- Список файлов, которые стоит подключать только на отдельных страницах
- Рекомендации по оптимизации:
  - разделение CSS (critical / page-specific)
  - lazy load JS
  - defer / async
  - объединение или наоборот разбиение файлов

6. Дополнительно:

- Если файл используется ТОЛЬКО на одной странице — явно указать это
- Если файл можно загружать динамически — предложить вариант (например через import())

7. ВАЖНО:
- Ничего не удалять автоматически
- Только анализ и рекомендации
- Быть максимально точным, не делать предположений без проверки

Результат должен быть максимально подробным и пригодным для внедрения.
